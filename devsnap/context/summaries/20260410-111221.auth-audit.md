# Authentication Security Audit

**Last Audit Date**: 2026-04-10
**Auditor**: Auth Security Agent

---

## Executive Summary

The authentication implementation is generally well-structured and correctly delegates session management, CSRF, and OAuth to NextAuth v5. However, the absence of any rate limiting across all custom auth endpoints is the most significant gap — it enables brute-force and email-bombing attacks with no friction. Several additional issues exist around session invalidation after password operations, a partial email enumeration leak in the forgot-password flow, and a missing maximum password length constraint that can be used for a denial-of-service via bcrypt CPU exhaustion.

## Findings

### Critical Issues

#### No Rate Limiting on Any Auth Endpoint

**Severity**: Critical
**Files**: All files under `src/app/api/auth/`
**Lines**: N/A (entire routes)

**Vulnerable Code**:
```typescript
// src/app/api/auth/register/route.ts
export async function POST(request: NextRequest) {
  // No rate limiting before bcrypt.hash(password, 12)
  const hashedPassword = await bcrypt.hash(password, 12)
  ...
}

// src/app/api/auth/forgot-password/route.ts
export async function POST(request: NextRequest) {
  // No rate limiting — sends unlimited emails
  await sendPasswordResetEmail(email, token)
  ...
}
```

**Problem**: Not a single custom auth endpoint has rate limiting. This affects:
- `/api/auth/register` — allows mass account creation (spam/abuse)
- `/api/auth/forgot-password` — allows email bombing any address with reset emails, one request per call to a third-party provider (Resend)
- `/api/auth/resend-verification` — same email bombing risk
- `/api/auth/change-password` — allows unlimited password-guessing attempts against a known current password, bypassing NextAuth's credential flow entirely
- `/api/auth/reset-password` — token guessing amplified if tokens had low entropy (they do not, but rate limiting is still required defense-in-depth)

Note: NextAuth handles rate limiting on its own `/api/auth/callback/credentials` path for the sign-in form. The custom routes above are outside that protection.

**Attack Scenario**:
1. Email bombing: An attacker submits `POST /api/auth/forgot-password` with a victim's email thousands of times per minute, flooding the victim's inbox and exhausting any Resend sending quota.
2. Registration spam: Bots create thousands of accounts per second, polluting the database and consuming resources.
3. Change-password brute force: An authenticated attacker whose account shares a device with the victim (or who has a stolen JWT) can script unlimited `POST /api/auth/change-password` attempts with different `currentPassword` values.

**Fix**: Add rate limiting to all custom auth endpoints. The simplest approach for a Vercel/Next.js deployment is `@upstash/ratelimit` with a Redis store, or use the `rate-limiter-flexible` package with an in-memory or Redis backend.

```typescript
// src/lib/rate-limit.ts
import { Ratelimit } from "@upstash/ratelimit"
import { Redis } from "@upstash/redis"

export const authRatelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, "1 m"), // 10 req/min per IP
  analytics: false,
})

// Usage in any route:
// src/app/api/auth/forgot-password/route.ts
import { authRatelimit } from "@/lib/rate-limit"

export async function POST(request: NextRequest) {
  const ip = request.headers.get("x-forwarded-for") ?? "unknown"
  const { success } = await authRatelimit.limit(ip)
  if (!success) {
    return NextResponse.json({ error: "Too many requests" }, { status: 429 })
  }
  // ... rest of handler
}
```

For the forgot-password and resend-verification endpoints specifically, also rate-limit by email address (not just IP) to prevent relay attacks through different IPs:

```typescript
// Rate limit by both IP and target email
const ipLimit = await authRatelimit.limit(`ip:${ip}`)
const emailLimit = await authRatelimit.limit(`email:${email}`)
if (!ipLimit.success || !emailLimit.success) {
  return NextResponse.json({ success: true }) // Silent — don't reveal limiting
}
```

### High Severity

#### No Password Maximum Length — bcrypt CPU Exhaustion (DoS)

**Severity**: High
**File**: `src/app/api/auth/register/route.ts`, `src/app/api/auth/reset-password/route.ts`, `src/app/api/auth/change-password/route.ts`
**Lines**: register:24-29, reset-password:17-19, change-password:18-20

**Vulnerable Code**:
```typescript
// register/route.ts — only minimum enforced, no maximum
if (password.length < 8) {
  return NextResponse.json({ error: "Password must be at least 8 characters" }, { status: 400 })
}

const hashedPassword = await bcrypt.hash(password, 12)
```

**Problem**: bcrypt truncates input at 72 bytes internally, but it must still process the full string before truncation — and `bcrypt.hash` is deliberately slow (12 rounds here). Submitting a password of several megabytes forces the server to run a CPU-intensive operation on a very large buffer. Without rate limiting this is a compounding DoS vector. The fix is to enforce a maximum length before hashing.

**Attack Scenario**: An attacker sends repeated registration or reset-password requests with 1 MB passwords. Each request pegs a CPU core for several hundred milliseconds. Combined with no rate limiting, this can exhaust server resources.

**Fix**: Add a maximum password length check before calling `bcrypt.hash` in all three routes:

```typescript
if (password.length < 8 || password.length > 72) {
  return NextResponse.json(
    { error: "Password must be between 8 and 72 characters" },
    { status: 400 }
  )
}
```

72 is the natural bcrypt limit; you can choose a lower practical limit (e.g. 128) as long as it is enforced server-side before hashing.

#### Active Sessions Not Invalidated After Password Change or Reset

**Severity**: High
**Files**: `src/app/api/auth/change-password/route.ts`, `src/app/api/auth/reset-password/route.ts`
**Lines**: change-password:43-46, reset-password:38-45

**Vulnerable Code**:
```typescript
// change-password/route.ts
await prisma.user.update({
  where: { id: session.user.id },
  data: { password: hashed },
})
return NextResponse.json({ success: true })
// No session invalidation

// reset-password/route.ts
await prisma.user.update({
  where: { email },
  data: { password: hashed },
})
await prisma.verificationToken.delete({ ... })
return NextResponse.json({ success: true })
// No session invalidation
```

**Problem**: The project uses JWT strategy (`session: { strategy: "jwt" }` in `src/auth.ts`). JWTs are stateless — they remain valid until expiry regardless of password changes. After a successful password reset (which is the primary account-recovery mechanism), any attacker who had a stolen session JWT can continue to use the account until the token naturally expires. After a password change by the legitimate user, any device where the old session is cached remains authenticated.

**Attack Scenario**: An attacker obtains a victim's JWT (e.g., from a compromised device, log, or XSS). The victim resets their password. The attacker's old JWT continues to grant full access to the account until it expires.

**Fix**: The cleanest approach with JWT strategy is to embed a `passwordHash` version or a `sessionVersion` counter in the user record, include it in the JWT at sign-in time, and validate it on every sensitive request.

```typescript
// prisma/schema.prisma — add to User model
sessionVersion Int @default(0)

// reset-password/route.ts — increment version on password change
await prisma.user.update({
  where: { email },
  data: {
    password: hashed,
    sessionVersion: { increment: 1 },
  },
})

// src/auth.ts — embed version in JWT
callbacks: {
  async jwt({ token, user, trigger }) {
    if (user?.id) {
      token.sub = user.id
      const dbUser = await prisma.user.findUnique({
        where: { id: user.id },
        select: { sessionVersion: true },
      })
      token.sessionVersion = dbUser?.sessionVersion ?? 0
    }
    return token
  },
  async session({ session, token }) {
    if (token.sub) session.user.id = token.sub
    return session
  },
},

// Then in change-password and any sensitive API route, verify:
const dbUser = await prisma.user.findUnique({
  where: { id: session.user.id },
  select: { sessionVersion: true },
})
if (dbUser?.sessionVersion !== token.sessionVersion) {
  return NextResponse.json({ error: "Session expired" }, { status: 401 })
}
```

Alternatively, for immediate revocation on password reset specifically: if you switch critical routes to database sessions, NextAuth will delete the session row on sign-out. For JWT strategy, the version approach above is the standard solution.

#### Email Enumeration via GitHub Account Detection in Forgot Password

**Severity**: High
**File**: `src/app/api/auth/forgot-password/route.ts`
**Lines**: 20-22

**Vulnerable Code**:
```typescript
// Unknown emails → silent success (correct)
if (!user) {
  return NextResponse.json({ success: true })
}

// GitHub-only accounts → distinct 400 error (leaks that email exists AND is OAuth-only)
if (!user.password) {
  return NextResponse.json({ error: "github-account" }, { status: 400 })
}
```

**Problem**: The code correctly returns a silent success for unknown emails to prevent enumeration. However, it returns a distinct error (`"github-account"`, HTTP 400) for emails that exist but have no password (GitHub OAuth users). This tells an attacker: "this email address is registered and uses GitHub OAuth." An attacker can enumerate which email addresses in a target list have DevSnap accounts and specifically which accounts use GitHub OAuth.

**Attack Scenario**: An attacker submits a list of email addresses to `/api/auth/forgot-password`. Any address that returns HTTP 400 with `"github-account"` is confirmed to be a registered user with GitHub OAuth. This information can be used for targeted phishing ("we noticed you use GitHub to sign in to DevSnap...").

**Fix**: Return the same silent success response for GitHub-only accounts, and display the "use GitHub" guidance only on the frontend based on a different signal (or handle it at sign-in time, which is already done there):

```typescript
// forgot-password/route.ts
const user = await prisma.user.findUnique({ where: { email } })

// Always succeed silently for both unknown emails AND GitHub-only accounts
if (!user || !user.password) {
  return NextResponse.json({ success: true })
}

// Continue with password reset only for credential users...
```

On the client, you can remove the GitHub-specific error branch since the server will never return it:

```typescript
// forgot-password/page.tsx — remove this branch
if (data.error === "github-account") {
  setError("This account uses GitHub to sign in. Password reset is not available.")
}
```

Users who have a GitHub account and try to reset a password will receive the "check your inbox" message but get no email — this is the correct behavior to avoid enumeration. The more helpful approach is to tell them at sign-in time (already implemented) rather than the reset form.

### Medium Severity

#### `crypto.randomUUID()` Tokens Stored in Plaintext — Token Theft from DB Read

**Severity**: Medium
**Files**: `src/app/api/auth/register/route.ts`, `src/app/api/auth/resend-verification/route.ts`, `src/app/api/auth/forgot-password/route.ts`
**Lines**: register:50, resend:23, forgot-password:30

**Vulnerable Code**:
```typescript
const token = crypto.randomUUID()
// ...
await prisma.verificationToken.create({
  data: { identifier: email, token, expires },
})
```

**Problem**: Verification and reset tokens are stored in plaintext in the `VerificationToken` table. If an attacker gains read access to the database (SQL injection elsewhere in the app, a database backup leak, or a misconfigured database connection), they can extract valid tokens and use them to verify arbitrary email addresses or reset any user's password directly. The token itself is cryptographically random, which is good, but storing it hashed adds a critical layer of defense-in-depth.

**Attack Scenario**: A database read is obtained (e.g., a future SQL injection vulnerability in item search, or an exposed backup). The attacker reads the `VerificationToken` table and finds an active reset token. They construct the reset URL and take over the account without knowing the current password or having email access.

**Fix**: Hash the token with SHA-256 before storing it. The raw token is sent in the email link; the hash is stored in the database. During verification, hash the submitted token and compare.

```typescript
import crypto from "crypto"

function hashToken(token: string): string {
  return crypto.createHash("sha256").update(token).digest("hex")
}

// When creating:
const rawToken = crypto.randomUUID()
const hashedToken = hashToken(rawToken)

await prisma.verificationToken.create({
  data: { identifier: email, token: hashedToken, expires },
})
// Send rawToken in the email URL

// When verifying (verify-email/route.ts, reset-password/route.ts):
const hashedToken = hashToken(submittedToken)
const record = await prisma.verificationToken.findUnique({
  where: { identifier_token: { identifier, token: hashedToken } },
})
```

#### No Input Validation on `name` Field at Registration

**Severity**: Medium
**File**: `src/app/api/auth/register/route.ts`
**Lines**: 16-18

**Vulnerable Code**:
```typescript
if (!name || !email || !password || !confirmPassword) {
  return NextResponse.json({ error: "All fields are required" }, { status: 400 })
}
// name is stored directly with no length or content validation
```

**Problem**: The `name` field has no length limit enforced server-side. A very long name (e.g., 100,000 characters) will be stored in the database and returned in JWT tokens on every request, inflating token size. There is also no sanitization, though since the field is stored and displayed in context where React handles encoding, XSS risk is low. The primary concern is the absence of a length cap.

**Fix**: Add a length check for `name` and validate `email` format server-side (the HTML `type="email"` constraint is client-side only and can be bypassed):

```typescript
import { z } from "zod"

const RegisterSchema = z.object({
  name: z.string().min(1).max(100),
  email: z.string().email().max(254),
  password: z.string().min(8).max(72),
  confirmPassword: z.string(),
})

// In the route:
const parsed = RegisterSchema.safeParse(body)
if (!parsed.success) {
  return NextResponse.json({ error: "Invalid input" }, { status: 400 })
}
```

#### Password Confirmation Sent Over the Wire Unnecessarily

**Severity**: Low
**File**: `src/app/(auth)/register/page.tsx`
**Lines**: 31-34

**Vulnerable Code**:
```typescript
body: JSON.stringify({ name, email, password, confirmPassword }),
```

**Problem**: The `confirmPassword` field is sent to the API route even though the client already validates that `password === confirmPassword` before submitting. The server also checks this (register/route.ts line 20-22), but there is no functional reason to transmit `confirmPassword` to the server — the server should only receive `password`. Transmitting `confirmPassword` doubles password exposure in request logs, proxies, or any network capture.

**Fix**: Remove `confirmPassword` from the request body. Keep the client-side equality check only. The server needs only `password`:

```typescript
// register/page.tsx
body: JSON.stringify({ name, email, password }), // Remove confirmPassword

// register/route.ts — remove confirmPassword from destructuring and check
const { name, email, password } = body
// Remove: if (password !== confirmPassword) ...
```

#### No Server-Side Email Format Validation on Auth Endpoints

**Severity**: Low
**Files**: `src/app/api/auth/forgot-password/route.ts`, `src/app/api/auth/resend-verification/route.ts`
**Lines**: forgot-password:7-10, resend:7-10

**Vulnerable Code**:
```typescript
const { email } = (await request.json()) as { email: string }

if (!email) {
  return NextResponse.json({ error: "Email is required" }, { status: 400 })
}
// email is used directly in a Prisma query with no format validation
```

**Problem**: These endpoints accept any string as `email` and pass it directly to Prisma. Prisma parameterizes queries, so SQL injection is not a concern. However, malformed email values (e.g., very long strings, strings without `@`) will hit the database unnecessarily. More importantly, there is no guarantee that the input conforms to an email format, which could lead to unexpected behavior in edge cases.

**Fix**: Add a basic email format check:

```typescript
import { z } from "zod"

const emailSchema = z.string().email().max(254)

const result = emailSchema.safeParse(email)
if (!result.success) {
  // Return same silent success to avoid information disclosure
  return NextResponse.json({ success: true })
}
```

## Passed Checks

The following security measures are correctly and explicitly implemented:

- **Password hashing with bcrypt at cost factor 12**: All three password-hashing sites (register, reset-password, change-password) use `bcrypt.hash(password, 12)`, which is a strong work factor above the minimum recommended 10.

- **Cryptographically secure token generation**: `crypto.randomUUID()` is used for all token generation. This is the Web Crypto API implementation and produces 122 bits of entropy — sufficient for unpredictable tokens.

- **Verification tokens are deleted after use**: `src/app/api/auth/verify-email/route.ts` correctly deletes the token after successful verification (single-use enforcement). The same applies in `reset-password/route.ts`.

- **Expired tokens are deleted on detection**: Both the verify-email and reset-password routes delete expired tokens when they are encountered, preventing accumulation of stale tokens.

- **Reset tokens have a short 1-hour expiry**: The forgot-password flow correctly creates tokens expiring in 1 hour, which is the recommended maximum for password reset tokens.

- **Old reset token is deleted before creating a new one**: `forgot-password/route.ts` calls `deleteMany` for any existing reset token before issuing a new one, preventing multiple valid reset tokens coexisting for the same account.

- **Change password uses session user ID, not request input**: `change-password/route.ts` reads `session.user.id` from the server-side session and queries the user by that ID — it never trusts a user-supplied ID in the request body.

- **Current password required to change password**: `change-password/route.ts` verifies the existing password with `bcrypt.compare` before allowing a change.

- **Unknown email enumeration prevented on forgot-password**: The route returns `{ success: true }` for email addresses not in the database, giving no confirmation of registration status.

- **Email verification checked before sign-in**: The Credentials `authorize` function in `src/auth.ts` throws `UnverifiedEmailError` if `emailVerified` is null, blocking unverified users from obtaining a session token.

- **Session validation on protected API routes**: `change-password/route.ts` and `delete-account/route.ts` both call `auth()` and check `session?.user?.id` before proceeding.

- **User ID from session, not from request body on delete-account**: `delete-account/route.ts` uses `session.user.id` for the `prisma.user.delete` call — an authenticated user can only delete their own account.

- **Cascade deletes configured in schema**: The Prisma schema has `onDelete: Cascade` on Account, Collection, Item, Tag, and ItemTag relations from User, ensuring account deletion removes all associated data.

- **Middleware protects `/dashboard` and `/profile` routes**: `src/proxy.ts` redirects unauthenticated requests on both route prefixes to sign-in, with the callback URL preserved.

- **Generic error message on sign-in failure**: The sign-in page displays "Invalid email or password." regardless of whether the email exists or the password is wrong — no enumeration.

- **SKIP_EMAIL_VERIFICATION is a dev-only escape hatch**: The flag is evaluated from `process.env` at runtime, and the production email flow is unaffected when the flag is absent.

- **bcryptjs is in production dependencies** (not devDependencies), ensuring hashing is available in production builds.

## Recommendations Summary (Priority Order)

1. **[Critical] Implement rate limiting on all custom auth endpoints** — use `@upstash/ratelimit` or equivalent. Endpoints to protect: register, forgot-password, resend-verification, change-password, reset-password. Target: 5-10 requests per minute per IP, plus per-email limits on email-sending endpoints.

2. **[High] Add a maximum password length check (72 characters) before bcrypt** — prevents CPU exhaustion DoS. Add to register, reset-password, and change-password routes.

3. **[High] Invalidate existing sessions after password change and reset** — introduce a `sessionVersion` integer on the User model, increment it on every password change/reset, embed it in the JWT, and validate it in protected routes. This prevents stolen JWTs from remaining valid after a password change.

4. **[High] Remove the GitHub account enumeration leak in forgot-password** — return the same silent `{ success: true }` for GitHub-only accounts as for unknown emails.

5. **[Medium] Hash tokens before storing in the VerificationToken table** — use SHA-256 to hash tokens at rest. Raw token goes in the email link; the hash is compared at verification time.

6. **[Medium] Add server-side input validation with Zod** — validate `name` length and `email` format on all auth endpoints. Remove `confirmPassword` from the register request body.

7. **[Low] Add email format validation to forgot-password and resend-verification** — prevents malformed strings from hitting the database.
